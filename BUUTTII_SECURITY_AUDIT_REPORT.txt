================================================================================
  1. EXECUTIVE SUMMARY
================================================================================

This audit identified 7 Critical, 6 High, 8 Medium, and 5 Low severity
findings across the Buuttii platform. The most dangerous issues are:

  - CRITICAL: Webhook has NO signature verification — any attacker can forge
    payment success and get free subscriptions (paymentController.js:360-391)
  - CRITICAL: Payment finalization has NO idempotency guard and NO transaction
    boundary — concurrent webhook+redirect causes double activation
  - CRITICAL: Meal skip extends subscription by raw calendar days, not meal
    days — users on no-Saturday plans get free meals
  - CRITICAL: SQL injection via string interpolation in meal skip queries
  - HIGH: .env file contains live PhonePe, Firebase, and Cloudinary secrets
    committed alongside code (even though .gitignore exists, secrets are real)
  - HIGH: Cart checkout uses stale cached prices, not re-validated at payment
  - HIGH: Admin analytics has SQL injection via schoolId interpolation

Immediate action is required on all Critical findings to prevent financial
loss and data corruption in production.

CRIT-01: PHONEPE WEBHOOK — ZERO AUTHENTICATION (PAYMENT FORGERY)
--------------------------------------------------------------------------------
Severity:       CRITICAL — Direct Financial Loss
File:           src/client/controllers/paymentController.js, lines 360-391
                src/client/routes/paymentRoutes.js, line 123

Trigger Scenario:
  Attacker sends POST to /api/client/payment/webhook with body:
  { "response": "<base64 of {success:true, code:'PAYMENT_SUCCESS',
    data:{merchantTransactionId:'TXN_123_xxx'}}>" }

Expected Behavior:
  Server validates HMAC/X-VERIFY header using PhonePe salt key before
  processing. Rejects unsigned or tampered payloads.

Actual Risky Behavior:
  handleWebhook() simply base64-decodes the payload, checks if
  code === 'PAYMENT_SUCCESS', and immediately calls
  finalizeSuccessfulPayment(). No signature check. No HMAC validation.
  The PhonePeUtil class HAS a validateCallback() method (phonepe.js:80-87)
  but it is NEVER called.

Technical Root Cause:
  Line 361: payload = req.body.response || req.body.request
  Line 366: decoded = JSON.parse(Buffer.from(payload, 'base64'))
  Line 375: status = (success && code === 'PAYMENT_SUCCESS') ? 'success'...
  Line 384: await finalizeSuccessfulPayment(txnResult.rows[0].order_id)
  — No call to phonepe.validateCallback() anywhere in the webhook handler.

Real Production Impact:
  ANY attacker who knows (or guesses) a merchantTransactionId can forge a
  webhook and activate subscriptions without paying. The merchantTransactionId
  format is predictable: TXN_{orderId}_{timestamp}.

Recommended Fix:
  BACKEND: Add to handleWebhook():
    const xVerify = req.headers['x-verify'];
    const isValid = phonepe.validateCallback(
      process.env.PHONEPE_WEBHOOK_USERNAME,
      process.env.PHONEPE_WEBHOOK_PASSWORD,
      xVerify,
      JSON.stringify(req.body)
    );
    if (!isValid) return res.status(401).send('Invalid Signature');

  DATABASE: Add processed_at TIMESTAMP column to transactions table.
  Check processed_at IS NULL before finalizing.

CRIT-02: NO IDEMPOTENCY — TRIPLE FINALIZATION RACE CONDITION
--------------------------------------------------------------------------------
Severity:       CRITICAL — Data Corruption / Double Activation
Files:          src/client/controllers/paymentController.js lines 150-187,
                360-391, 399-439, 448-532, 747-796

Trigger Scenario:
  PhonePe fires webhook (line 384) and redirect callback (line 426) within
  milliseconds of each other. User also hits force-sync (line 787).
  All three call finalizeSuccessfulPayment() for the same orderId.

Expected Behavior:
  Only ONE execution should activate the subscription. Subsequent calls
  should be no-ops (idempotent).

Actual Risky Behavior:
  finalizeSuccessfulPayment() (line 150):
    1. UPDATE orders SET status='completed' — no WHERE status='pending' guard
    2. Reads order, loops through cart items
    3. Calls activateSingleSubscription() for each
  
  activateSingleSubscription() (line 93):
    - Reads existingSub (line 103) without FOR UPDATE lock
    - Calculates newTotalMeals = oldTotal + computeMealCount() (line 128)
    - UPSERT via ON CONFLICT (line 136-144)
  
  With two concurrent executions:
    Thread A reads oldTotal=30, computes newTotalMeals=30+26=56
    Thread B reads oldTotal=30, computes newTotalMeals=30+26=56
    Both UPSERT → last writer wins, but total_meals=56 instead of
    correct 30+26=56 (lucky) or could be 30+26+26=82 (if stacking)

  The redirect callback (line 417) checks `status='pending'` for the
  transaction but NOT for the order. The webhook (line 377) doesn't
  check either — it updates transaction status unconditionally.

Technical Root Cause:
  - No database transaction wrapping finalizeSuccessfulPayment()
  - No SELECT FOR UPDATE on client_subscriptions
  - No idempotency key or processed_at check
  - Order status updated BEFORE activation, not AFTER in same txn

Real Production Impact:
  - Corrupted total_meals counts
  - Double subscription activation for cart orders
  - Payment marked completed but some entities not activated (partial commit)

Recommended Fix:
  BACKEND:
    const client = await db.pool.connect();
    await client.query('BEGIN');
    // Lock the order row
    const orderLock = await client.query(
      'UPDATE orders SET status=$1, updated_at=NOW() WHERE id=$2 AND status=$3 RETURNING *',
      ['completed', orderId, 'pending']
    );
    if (orderLock.rowCount === 0) { await client.query('ROLLBACK'); return; }
    // ... activate subscriptions using client (same txn) ...
    // Use SELECT ... FOR UPDATE on client_subscriptions
    await client.query('COMMIT');

  DATABASE:
    ALTER TABLE transactions ADD COLUMN processed_at TIMESTAMPTZ;
    — Only finalize when processed_at IS NULL, set it atomically.

CRIT-03: SQL INJECTION VIA STRING INTERPOLATION IN MEAL SKIP
--------------------------------------------------------------------------------
Severity:       CRITICAL — SQL Injection
Files:          src/client/controllers/mealController.js lines 325-329, 405-410

Trigger Scenario:
  mealController.js line 327:
    SET end_date = end_date + INTERVAL '${totalDays} days'
  
  mealController.js line 407:
    SET end_date = end_date - INTERVAL '${skipData.total_skip_days} days'

  totalDays on line 327 is computed in JS as an integer (line 260), so
  direct injection from request is unlikely. HOWEVER, skipData.total_skip_days
  on line 407 is READ FROM THE DATABASE (line 397: skip.rows[0]).

  If the total_skip_days column ever contains a corrupted value (e.g., via
  direct DB edit, migration bug, or a future code path), it becomes a
  second-order SQL injection vector.

Expected Behavior:
  All SQL should use parameterized queries exclusively.

Actual Risky Behavior:
  Template literal interpolation inside SQL string. Even though the current
  code path computes totalDays as an integer, the PATTERN is unsafe and
  violates defense-in-depth.

Recommended Fix:
  Replace:
    SET end_date = end_date + INTERVAL '${totalDays} days'
  With:
    SET end_date = end_date + ($X * INTERVAL '1 day')
  where $X is a parameterized integer value.

CRIT-04: MEAL SKIP EXTENSION BUG — FREE MEALS EXPLOIT
--------------------------------------------------------------------------------
Severity:       CRITICAL — Business Logic / Financial Loss
File:           src/client/controllers/mealController.js lines 317-331

Trigger Scenario:
  User has "without Saturday" plan (include_saturday=false).
  User requests skip for Fri-Sat-Sun (3 days, meets minimum).
  System extends end_date by 3 calendar days.
  But Saturday was never a meal day — user gets 2 free weekday meals.

  Worse: User can skip Sat-Sun-Mon (3 days). Only Monday was a meal day.
  System extends by 3 days = 2 free meal-days gained.

Expected Behavior:
  Extension should count only actual meal days within the skip range.
  For a no-Saturday plan skipping Fri-Sat-Sun, extension = 2 days (Fri+Sun).

Actual Risky Behavior:
  Line 260: totalDays = daysBetweenInclusive(startYmd, endYmd)  // raw calendar
  Line 327: end_date = end_date + INTERVAL '${totalDays} days'  // extends by calendar days
  
  No consideration of include_saturday flag on the subscription.
  No filtering of non-meal days from the skip range.

Technical Root Cause:
  The skip validation checks the subscription date range (lines 301-306)
  but never consults include_saturday. The extension is always raw days.

Real Production Impact:
  Users can systematically exploit this to extend subscriptions beyond
  what they paid for. A 30-day plan could be extended to 35+ days
  by strategically placing skips over weekends.

Recommended Fix:
  BACKEND: Count only meal-eligible days in the skip range:
    const sub = subCheck.rows[0];
    let mealDaysSkipped = 0;
    for (let i = 0; i < totalDays; i++) {
      const d = addDaysYmd(startYmd, i);
      const isSat = new Date(d).getUTCDay() === 6;
      if (sub.include_saturday || !isSat) mealDaysSkipped++;
    }
    // Extend by mealDaysSkipped, not totalDays

CRIT-05: ADMIN ANALYTICS — SQL INJECTION VIA schoolId
--------------------------------------------------------------------------------
Severity:       CRITICAL — SQL Injection
File:           src/admin/controllers/subscriptionAnalyticsController.js

Trigger Scenario:
  Line 444: if (schoolId) childQuery += ` AND ch.school_id = '${schoolId}'`;
  Line 611: ${schoolId ? `AND ch.school_id = '${schoolId}'` : ''}

  Admin sends: GET /api/admin/subscriptions/analytics/all-members?schoolId='; DROP TABLE clients; --

Expected Behavior:
  schoolId should be parameterized.

Actual Risky Behavior:
  Direct string interpolation of user-supplied schoolId into SQL.
  Also on line 549: LIMIT ${parseInt(limit)} OFFSET ${parseInt(offset)}
  — parseInt provides some protection but the pattern is still risky.

Real Production Impact:
  Admin-authenticated SQL injection. An attacker with admin credentials
  (or a compromised admin session) can execute arbitrary SQL.

Recommended Fix:
  Use parameterized queries for all user-supplied values.

CRIT-06: ADMIN ANALYTICS — SQL INJECTION VIA days PARAMETER
--------------------------------------------------------------------------------
Severity:       CRITICAL — SQL Injection
File:           src/admin/controllers/subscriptionAnalyticsController.js line 390

Code:           AND cs.end_date <= NOW() + ($1 || ' days')::INTERVAL

Trigger Scenario:
  GET /api/admin/subscriptions/analytics/expiring-soon?days=1' OR '1'='1

  The $1 parameter is parseInt(days) on line 355, which provides protection
  IF parseInt returns NaN for malicious input (it would return NaN, then
  PostgreSQL would error). However, the pattern of concatenating user input
  into an INTERVAL string is inherently risky.

Recommended Fix:
  Use: AND cs.end_date <= NOW() + ($1::int * INTERVAL '1 day')

  3. HIGH SEVERITY FINDINGS

--------------------------------------------------------------------------------
HIGH-01: NON-ATOMIC PAYMENT FINALIZATION — PARTIAL COMMIT RISK
--------------------------------------------------------------------------------
Severity:       HIGH
File:           src/client/controllers/paymentController.js lines 150-187

The function finalizeSuccessfulPayment() runs multiple DB operations:
  1. UPDATE orders SET status='completed' (line 152)
  2. For each cart item: activateSingleSubscription() (lines 160-170)
  3. UPDATE carts SET status='checked_out' (line 172)

These are NOT wrapped in a transaction. If the server crashes after step 1
but before step 2 completes, the order is 'completed' but subscriptions
are not activated. The user paid but gets nothing. Subsequent force-sync
calls would try to finalize again but order is already 'completed'.

Fix: Wrap everything in BEGIN/COMMIT with a connection from pool.connect().

--------------------------------------------------------------------------------
HIGH-02: STALE CART PRICING — REVENUE LEAKAGE
--------------------------------------------------------------------------------
Severity:       HIGH
File:           src/client/controllers/paymentController.js lines 283-352

checkoutCart() reads total_amount directly from the carts table (line 301):
  const totalAmount = parseFloat(cart.total_amount);

This total was calculated when items were ADDED to the cart (cartController.js
line 92-96). If an admin changes plan prices between cart-add and checkout,
the user pays the OLD price.

No re-validation of:
  - Plan prices against current subscriptions table
  - Plan is_active status
  - Entity ownership (entities could have been transferred/deleted)

Fix: Re-fetch all cart_items, join with subscriptions, recalculate total,
and validate each entity during checkout.

--------------------------------------------------------------------------------
HIGH-03: SUBSCRIPTION UPSERT DESTROYS HISTORY
--------------------------------------------------------------------------------
Severity:       HIGH
File:           src/client/controllers/paymentController.js lines 136-144

ON CONFLICT (client_id, entity_id, entity_type) DO UPDATE overwrites:
  - start_date, end_date, subscription_id, order_id, total_meals, used_meals

This means if a user buys Plan A, uses 10 meals, then buys Plan B:
  - The old subscription record is completely overwritten
  - Historical audit trail is lost
  - If activateSingleSubscription runs for Plan A's order after Plan B,
    Plan B's data gets overwritten back to Plan A

Fix: Use a subscription_history/ledger table. Or add order_id to the
unique constraint so each purchase creates a separate row.

--------------------------------------------------------------------------------
HIGH-04: CONCURRENT MEAL SKIP — DOUBLE EXTENSION
--------------------------------------------------------------------------------
Severity:       HIGH
File:           src/client/controllers/mealController.js lines 308-331

The overlap check (line 309-315) uses a standard SELECT, not SELECT FOR UPDATE.
Two concurrent skip requests for adjacent/overlapping ranges can both pass
the overlap check and both extend the end_date.

Example:
  Request A: skip May 10-12 (3 days) → checks overlap → none found → extends
  Request B: skip May 11-13 (3 days) → checks overlap → none found (A not yet committed) → extends
  Result: end_date extended by 6 days, but only 5 unique days were skipped.

Fix: Use SELECT ... FOR UPDATE on the subscription row, or add a UNIQUE
constraint on (client_id, entity_type, entity_id, skip_start_date).

--------------------------------------------------------------------------------
HIGH-05: SKIP CANCELLATION — END_DATE CORRUPTION
--------------------------------------------------------------------------------
Severity:       HIGH
File:           src/client/controllers/mealController.js lines 387-418

cancelMealSkip() subtracts skip days from end_date (line 405-410):
  SET end_date = end_date - INTERVAL '${skipData.total_skip_days} days'

Problem scenarios:
  1. User creates Skip A (5 days, extends end_date by 5)
  2. User creates Skip B (3 days, extends end_date by 3)
  3. User cancels Skip A → subtracts 5 from current end_date
  4. But current end_date includes Skip B's extension
  5. Result: end_date is now LESS than what Skip B's extension requires

  Also: If subscription was renewed between skip creation and cancellation,
  the subtraction applies to the NEW subscription's end_date, potentially
  reducing it below the start_date.

Fix: Store the original end_date before each skip extension. On cancellation,
recalculate end_date from the base date plus remaining approved skips.

--------------------------------------------------------------------------------
HIGH-06: CORS WIDE OPEN
--------------------------------------------------------------------------------
Severity:       HIGH
File:           src/server.js line 65

  app.use(cors());

No origin whitelist. Any website can make authenticated requests to the API
if the user has a valid token (stored in localStorage/memory).

Fix: Specify allowed origins:
  app.use(cors({ origin: ['https://buuttii.com', 'https://admin.buuttii.com'] }));

  4. MEDIUM SEVERITY FINDINGS

MED-01: IDOR — resolveEntityName() has no ownership check
  File: paymentController.js line 10-16, cartController.js line 27-41
  resolveEntityName() queries by entityId without checking ownership.
  While used internally, if exposed it leaks names of other users' children.

MED-02: Pagination injection — limit/offset not sanitized
  File: paymentController.js line 656-680
  const { page = 1, limit = 10 } = req.query;
  Used directly in SQL LIMIT $2 OFFSET $3 without parseInt validation.
  Could cause unexpected behavior with non-numeric values.

MED-03: Teacher/Professional profile deletion allows admin impersonation
  File: teacherController.js line 165, professionalController.js line 167
  If req.user.role === 'admin', uses req.query.clientId. No validation
  that the clientId is a valid client. Admin could pass arbitrary values.

MED-04: No rate limiting on webhook endpoint
  File: server.js lines 86-87
  Rate limiter only applies to /api/client and /api/common.
  The webhook at /api/client/payment/webhook IS rate-limited, but
  /api/admin routes have NO rate limiting at all.

MED-05: Open redirect via frontendUrl parameter
  File: paymentController.js line 402, 438
  frontendUrl comes from query parameter. An attacker could craft a callback
  URL that redirects users to a phishing site after payment.

MED-06: JWT token contains role in payload — privilege escalation risk
  File: client/middlewares/authMiddleware.js line 12-18
  Token payload includes { id, phoneNumber, role: 'client' }.
  If CLIENT_JWT_SECRET is weak (it IS — see CRIT-07), an attacker could
  forge a token with role:'admin' and use it against client endpoints.
  Note: admin and client use DIFFERENT secrets, so cross-role forgery
  requires knowing both. But the secrets are trivially guessable.

MED-07: No Swagger authentication — API docs exposed
  File: server.js line 90
  app.use('/api-docs', swaggerUi.serve, swaggerUi.setup(specs));
  No auth middleware. Full API documentation is publicly accessible.

MED-08: Flutter cart provider — stale state after payment
  File: lib/core/providers/cart_provider.dart lines 311-316
  Comment explicitly says "do NOT clear local cart here" after SDK SUCCESS.
  But no automatic re-fetch is triggered either. The subscription provider
  and meal provider are not refreshed after payment completion.

================================================================================
  6. EXPLOIT SCENARIOS (DETAILED)
================================================================================

EXPLOIT 1: Free Subscription via Webhook Forgery
  1. Attacker registers account, adds child, initiates payment
  2. Copies merchantTransactionId from payment response
  3. Cancels/abandons the actual PhonePe payment
  4. Crafts POST to /api/client/payment/webhook with forged success payload
  5. System activates subscription without any real payment
  Time to exploit: ~2 minutes
  Financial impact: Unlimited free subscriptions

EXPLOIT 2: Free Meals via Skip Manipulation
  1. User has 30-day plan without Saturday (include_saturday=false)
  2. Every week, user skips Fri-Sat-Sun (3 days, meets minimum)
  3. System extends end_date by 3 calendar days each time
  4. But Saturday was never a meal day — net gain: 1 free day per skip
  5. Over 4 weeks: 4 extra meal days = ~13% free meals
  Financial impact: ~13% revenue leakage per no-Saturday subscriber

EXPLOIT 3: Double Payment Activation
  1. User completes payment on PhonePe
  2. Redirect callback fires, calls finalizeSuccessfulPayment()
  3. Webhook fires 200ms later, calls finalizeSuccessfulPayment() again
  4. Both read existing subscription, both add meal counts
  5. total_meals gets doubled (60 instead of 30 for a 30-day plan)
  Financial impact: 2x meals delivered per payment

EXPLOIT 4: Stale Price Checkout
  1. User adds 30-day plan at Rs.800 to cart
  2. Admin raises price to Rs.1000
  3. User checks out → pays Rs.800 (cached in cart_items.unit_price)
  4. Subscription activated for the Rs.1000 plan at Rs.800 price
  Financial impact: Per-transaction revenue loss


================================================================================
  7. DATABASE RISKS
================================================================================

- client_subscriptions has no CHECK constraint on used_meals <= total_meals
  → used_meals can theoretically go negative or exceed total_meals

- No partial unique index for active subscriptions
  → ON CONFLICT overwrites history, no audit trail

- meal_skips has no unique constraint preventing overlapping date ranges
  → Concurrent requests can create overlapping skips

- No foreign key constraint between meal_skips.entity_id and the actual
  entity tables (children, teacher_profiles, professional_profiles)
  → Orphaned skip records possible after entity deletion

- daily_meal_log UNIQUE(subscription_id, meal_date) prevents double-counting
  per day — this is GOOD and correctly implemented


================================================================================
  8. CONCURRENCY RISKS
================================================================================

- activateSingleSubscription: No row-level locking. Concurrent renewals for
  same entity corrupt total_meals. Need SELECT FOR UPDATE.

- requestMealSkip: Overlap check without row lock. Two concurrent skip
  requests can both pass validation and both extend end_date.

- executeMealReductionForDate: CORRECTLY uses BEGIN/COMMIT and FOR UPDATE.
  This is the BEST-IMPLEMENTED transactional code in the codebase.

- cancelMealSkip: No transaction boundary. Subscription update and skip
  status update are separate queries — partial commit possible.

- checkoutCart: No locking on cart. Theoretically, two concurrent checkouts
  of the same cart could create two orders for the same cart.


================================================================================
  9. AUTHORIZATION RISKS
================================================================================

- Entity ownership is correctly validated in: initiatePayment, addToCart,
  requestMealSkip, addChildren, updateChild, deleteChild ✓

- Entity ownership is NOT validated in: resolveEntityName (info leak),
  resolveEntityMealMeta (minor, internal use)

- Admin APIs have no school/tenant scoping — any admin sees all data.
  If multi-admin is added later, this becomes a horizontal privilege issue.

- checkPaymentStatus (line 478): Checks client_id match only if
  req.user.role === 'client'. If role check is bypassed (weak JWT secret),
  any user can view any transaction.

  11. RECOMMENDED DATABASE CONSTRAINTS
================================================================================

  ALTER TABLE client_subscriptions
    ADD CONSTRAINT chk_meals CHECK (used_meals >= 0 AND used_meals <= total_meals);

  CREATE UNIQUE INDEX IF NOT EXISTS uniq_active_sub
    ON client_subscriptions (client_id, entity_id, entity_type)
    WHERE is_active = true;

  ALTER TABLE meal_skips
    ADD CONSTRAINT chk_skip_dates CHECK (skip_end_date >= skip_start_date);

  ALTER TABLE meal_skips
    ADD CONSTRAINT chk_skip_days CHECK (total_skip_days > 0);

  -- Prevent overlapping approved skips at DB level:
  CREATE EXTENSION IF NOT EXISTS btree_gist;
  ALTER TABLE meal_skips
    ADD CONSTRAINT no_overlap_skips
    EXCLUDE USING gist (
      entity_id WITH =,
      entity_type WITH =,
      daterange(skip_start_date, skip_end_date, '[]') WITH &&
    ) WHERE (status = 'approved');


================================================================================
  12. RECOMMENDED TRANSACTION BOUNDARIES
================================================================================

  MUST be transactional (currently NOT):
    - finalizeSuccessfulPayment() — order update + subscription activation
    - cancelMealSkip() — subscription update + skip status update
    - requestMealSkip() — overlap check + skip insert + subscription update
    - checkoutCart() — price revalidation + order creation + txn creation

  Already correctly transactional (good):
    - executeMealReductionForDate() ✓
    - reverseMealReductionForDate() ✓
    - adminAddExtraMealsByEntity() ✓
    - addChildren() ✓


================================================================================
  13. RECOMMENDED LOCKING STRATEGIES
================================================================================

  activateSingleSubscription():
    SELECT ... FROM client_subscriptions
    WHERE client_id=$1 AND entity_id=$2 AND entity_type=$3
    FOR UPDATE
    → Prevents concurrent renewal race conditions

  requestMealSkip():
    SELECT ... FROM client_subscriptions
    WHERE client_id=$1 AND entity_type=$2 AND entity_id=$3
    FOR UPDATE
    → Prevents concurrent skip creation

  checkoutCart():
    SELECT ... FROM carts WHERE id=$1 AND status='active'
    FOR UPDATE
    → Prevents double checkout of same cart


================================================================================
  14. FLUTTER STATE CONSISTENCY RISKS
================================================================================

  - PaymentProvider.initiateCheckout() does not trigger refresh of
    SubscriptionProvider or MealProvider after success
  - CartProvider.checkoutAll() explicitly does NOT clear cart on SDK success
    (correct decision), but no automatic polling/refresh is implemented
  - No global "post-payment refresh" coordinator exists
  - If session expires during PhonePe redirect flow, user returns to
    login screen and loses transaction context
  - _autoCorrectInvalidStartDates() silently changes cart item dates
    without user notification — could cause confusion

  Recommended:
    - Add a PaymentCompletionCoordinator that listens for payment success
      and triggers: cartProvider.resetLocal(), subscriptionProvider.refresh(),
      mealProvider.refresh(), paymentProvider.fetchActiveSubscriptions()
    - Implement a "pending transaction recovery" screen on app resume
